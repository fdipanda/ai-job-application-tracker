AI Job Application Tracker

Mentor walkthrough of the project for a senior Computer Science student with an AI concentration.

1. Project Overview

This project is a full-stack job application tracker with two input paths: manual entry and automatic extraction from Outlook email. The goal is to keep a single source of truth for applications and reduce the manual work of updating statuses as recruiters and ATS systems send messages.

At a high level, the architecture is:

  • React/Next.js frontend for the dashboard UI
  • FastAPI backend for REST endpoints and email ingestion
  • SQLite database for persistence
  • Microsoft Graph API for reading Outlook mail
  • OpenAI plus regex parsing for extracting job details from emails

The runtime flow is:

  • The frontend loads the dashboard from frontend/src/app/page.tsx.
  • It calls the backend through frontend/src/lib/api.ts.
  • The backend starts in backend/app/main.py, mounts CRUD routes from backend/app/routes.py, and creates tables from models in backend/app/models.py.
  • Application records are stored in SQLite using SQLAlchemy configured in backend/app/database.py.
  • If email sync is triggered, the backend authenticates with Microsoft, fetches messages, filters likely job mail, parses details, deduplicates by a generated job_key, then inserts or updates rows.

The code is organized around layers:

  • API layer: main.py, routes.py
  • Data layer: database.py, models.py, schemas.py
  • Business logic: services/application_service.py
  • Email pipeline: email_service.py, job_detector.py, ai_email_parser.py, email_parser.py, email_classifier.py

2. Technology Breakdown

  • Python: backend language. Good for APIs, automation, AI integrations, and quick iteration. Trade-off: not as strong as C# for compile-time guarantees.
  • FastAPI: modern Python web framework. It gives route definitions, validation, JSON serialization, and auto docs with little boilerplate. Trade-off: smaller batteries-included ecosystem than Django.
  • React plus Next.js: frontend stack. React handles component-based UI; Next.js provides project structure, bundling, dev server, and deployment-friendly conventions. Trade-off: more JavaScript ecosystem complexity.
  • TypeScript: adds static typing to frontend code. Useful for reducing UI and API mismatch bugs.
  • SQLite: embedded relational database stored in a file. Perfect for a small local app with simple deployment. Trade-off: limited concurrency and horizontal scaling.
  • SQLAlchemy: ORM for Python. It maps classes to database tables and hides a lot of raw SQL. Trade-off: abstraction can make debugging generated queries harder.
  • REST APIs: the frontend talks to the backend over HTTP using JSON. Simple and standard for web apps.
  • Microsoft Graph API: lets the app read Outlook messages after OAuth login. This is how email ingestion works.
  • OAuth plus MSAL: handles Microsoft sign-in and access tokens. Important because the app should never directly handle your Microsoft password.
  • OpenAI API: used in backend/app/services/ai_email_parser.py to extract company, role, location, and status from messy email text.
  • Bash and terminal tools: used for setup, running servers, tests, and dependency management.
  • Git: version control for tracking changes and collaborating safely.

3. Key Design Decisions

  • Python backend: likely chosen because this project mixes API work, text parsing, and AI integration. Python is especially strong there.
  • FastAPI over Flask or Django: FastAPI gives cleaner typing and validation than Flask, while staying lighter than Django. Good middle ground.
  • React and Next.js frontend: strong ecosystem, reusable components, and industry relevance. It also teaches modern frontend patterns.
  • SQLite over PostgreSQL or MySQL: fastest way to get persistence without running a separate database server.
  • SQLAlchemy ORM: helpful because the app has structured relational data but still benefits from Python objects and query abstraction.
  • Service-layer business logic: routes stay thin, while logic lives in services like backend/app/services/application_service.py. That is a good maintainability choice.
  • Terminal-heavy workflow: common because it is scriptable, reproducible, fast, and works the same on local machines and servers.

4. Alternative Implementations

Backend alternatives

  • C#: you would likely use ASP.NET Core plus Entity Framework. Strong typing, excellent tooling, great for larger enterprise systems.
  • Java: probably Spring Boot plus JPA or Hibernate. Very production-proven, but heavier and more verbose.
  • Node.js: Express or NestJS would keep frontend and backend both in JavaScript or TypeScript. Nice stack consistency, but less pleasant for some data and AI tasks.
  • Go: simpler deployment and strong performance, but less ergonomic for rapid AI or text-processing work.

Frontend alternatives

  • Angular: more opinionated, bigger framework, great structure, steeper learning curve.
  • Vue: often easier to learn than React, very productive for small and medium apps.
  • Svelte: very clean developer experience, less ecosystem depth than React.
  • Server-rendered HTML: simpler for small apps, but less dynamic for an interactive dashboard.

Database alternatives

  • PostgreSQL: best next step for production. Better concurrency, indexing, extensions, and reliability.
  • MySQL: similar category, widely used.
  • MongoDB: useful if the data were document-shaped and schema-flexible, less ideal here because applications fit relational structure well.
  • Redis: not a primary database here; better as a cache or queue helper.

ORM alternatives

  • Raw SQL: maximum control, more manual work.
  • Prisma: common in TypeScript ecosystems, not Python.
  • Django ORM: excellent if the backend were Django.
  • Entity Framework: the closest C# analogy to SQLAlchemy.

ORMs exist because they map tables to objects, reduce repetitive SQL boilerplate, centralize schema logic, and improve developer speed for common CRUD tasks.

5. Development Workflow

Typical local workflow:

  • Backend setup: cd backend, create a virtual environment, pip install -r requirements.txt, run uvicorn app.main:app --reload
  • Frontend setup: cd frontend, npm install, npm run dev
  • Tests: ./venv/bin/pytest -q

Why terminal workflows matter:

  • They are fast once learned
  • They can be automated in scripts and CI
  • They work consistently across local, server, and cloud environments
  • They expose the real commands that build and run software

Compared to GUI tools, the terminal has a steeper learning curve, but much more automation power.

6. What Could Be Improved

  • Auth persistence and security: access tokens are stored in process memory and in SQLite, which is convenient but not ideal for production.
  • Pydantic v2 cleanup: tests show deprecated dict() and orm_mode usage.
  • Better validation: statuses and input fields could be more strictly constrained.
  • Logging and observability: current logging is basic and local-file oriented.
  • Background jobs: email sync currently runs inline in request handlers.
  • More tests: backend is well covered, but frontend tests are missing.
  • Config separation: environment-specific config could be cleaner.

7. Scaling the System

If this needed thousands or millions of users:

  • Replace SQLite with PostgreSQL
  • Put backend and frontend in containers with Docker
  • Deploy behind load balancers
  • Move email sync to background workers
  • Add a queue like RabbitMQ, SQS, or Redis-based workers
  • Store secrets in a proper secret manager
  • Use caching for expensive reads
  • Add CI/CD pipelines
  • Consider splitting services only if the monolith becomes a bottleneck

Right now this is a good small monolith. At scale, the ingestion pipeline is the first thing I would separate operationally.

8. Industry Best Practices

A production-grade version would usually include:

  • Unit, integration, and frontend tests
  • Pull requests and code review
  • Linting, formatting, and type checks in CI
  • Structured logs, metrics, and tracing
  • Secret management with cloud tools, not only .env
  • Staging and production environments
  • Safer auth and session handling
  • Migrations for schema changes instead of implicit table creation

9. Mini Practice Projects

  • Build one FastAPI endpoint that returns a list of books. Teaches routing, JSON responses, and request handling.
  • Build one React component with a search box filtering a list. Teaches state, props, and rendering.
  • Build a tiny Python script that inserts and reads rows from SQLite. Teaches persistence basics.
  • Write a bash script that starts backend and frontend dev servers. Teaches terminal automation.
  • Call a public REST API from Python or JavaScript and display the result. Teaches HTTP, JSON, and async flow.

10. Using AI Tools for Development

AI tools can help a lot if used intentionally:

  • ChatGPT or Codex: good for explaining unfamiliar files, generating starter code, and comparing architecture options
  • Cursor or Copilot: great for inline code completion and refactors
  • Agents: useful for repetitive tasks like wiring CRUD or writing tests

Good prompts are specific:

  • Explain this file as if I know C# but not FastAPI
  • Trace the data flow from this button click to the database
  • Write tests for this service and explain the edge cases
  • Compare this SQLAlchemy model to the Entity Framework equivalent

Best practice is to use AI to accelerate understanding and drafting, verify important code paths yourself, ask it to explain trade-offs, and treat it like a strong junior pair programmer rather than an infallible architect.
